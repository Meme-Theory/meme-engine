# Knowledge Database — Schema-Driven Specification

The knowledge database is the project's institutional memory — a structured, queryable graph of everything established across sessions. This document specifies how to define, maintain, and query it without domain-specific scripts.

**Design principle**: The indexer agent IS the entity extractor. An LLM reading session artifacts doesn't need regex patterns — it needs a schema that tells it what to look for, what fields to fill, and what authority hierarchy to follow.

**What this replaces**: The 2,296-line `extract_entities.py` with 100+ hardcoded physics regex patterns. The schema-driven approach eliminates the need for domain-specific extraction scripts. Python tools (`knowledge_db.py`, `visualize_knowledge.py`) remain as optional accelerators for large projects.

---

## 1. Architecture

### 1a. Core Architecture (No Python Required)

```
sessions/*.md              ─┐
data/output/*               ─┼─▶  INDEXER AGENT  ─▶  knowledge-index.json
constraint-map.md           ─┘    (reads schema)           │
                                                            │
                              knowledge-schema.yaml ────────┘
                              (defines entity types,
                               fields, authority)
```

The indexer reads session artifacts, consults the schema for what entity types to extract and what fields are required, and writes entries to `knowledge-index.json`. No scripts. No regex. No Python dependency.

### 1b. Accelerated Architecture (Optional Python)

For projects that grow beyond ~10 sessions or ~1,000 entities, add the query accelerator layer:

```
knowledge-index.json (canonical)
        │
        ├──▶ knowledge_db.py ──▶ knowledge.db (SQLite + FTS5 ranked search)
        └──▶ visualize_knowledge.py ──▶ tools/viz/*.png + .mmd
```

These Python tools are **optional**. They consume the JSON index (which the indexer maintains) and produce derived artifacts. They are never required for the core knowledge system to function.

### 1c. What Lives Where

| Artifact | Location | Owner | Purpose |
|:---------|:---------|:------|:--------|
| Schema definition | `tools/knowledge-schema.yaml` | Generated during unfold | Defines entity types, fields, authority |
| Knowledge index | `tools/knowledge-index.json` | Indexer (sole writer) | Canonical knowledge graph |
| SQLite cache | `tools/knowledge.db` | Optional accelerator | FTS5 ranked search |
| Visualizations | `tools/viz/` | Optional accelerator | PNGs + Mermaid diagrams |
| Constraint map | `.claude/agent-memory/coordinator/constraint-map.md` | Coordinator (sole writer) | Logic backbone (see constraint-map.md spec) |

## 2. Knowledge Schema Format

### 2b. Field Type Reference

| Type | Meaning | JSON Representation |
|:-----|:--------|:--------------------|
| `string` | Text | `"value"` |
| `string[]` | List of strings | `["a", "b"]` |
| `number` | Numeric value | `42` |
| `date` | ISO 8601 date | `"2026-02-28"` |
| `enum` | One of listed values | `"PASS"` |
| `boolean` | True/false | `true` |

### 2c. Schema Validation Rules

1. Every entity type must have at least one `required: true` field
2. Every entity type must have a `source_file` field (provenance tracking)
3. Every entity type with `authority` must name an archetype that exists in the project
4. Enum fields must have a non-empty `values` list
5. Pattern fields use `{PLACEHOLDER}` notation (validated by the indexer, not enforced by regex)
6. Universal types cannot be removed — they can be renamed via `display_name` but must retain their structural role

## 3. Domain Adaptation Protocol

During `/new-research-project` unfolding, the indexer configures the schema for the target domain. This is the critical "flexible knowledge types" mechanism.

### 3a. What the Indexer Does

1. **Reads the domain** from coordinator or user input (e.g., "computational biology", "algebraic topology")
2. **Keeps all universal types** — these are non-negotiable infrastructure
3. **Renames where appropriate** — `proven_results` might display as "Findings" in social science or "Validated Models" in ML
4. **Adds domain-specific entity types** — the 10th+ type that captures domain-specific structured content
5. **Configures constraint categories** — adds domain-specific prefixes beyond S/F/D/O
6. **Writes the schema** to `tools/knowledge-schema.yaml`

### 3b. Domain-Specific Entity Type Examples

The 9th universal type slot (`equations` in the origin project) is the primary domain adaptation point. Projects replace it with whatever structured content matters most in their domain:

| Domain | Entity Type | Fields | What It Captures |
|:-------|:------------|:-------|:-----------------|
| **Theoretical Physics** | `equations` | id, raw, latex, type, context, name, audit_status | Named mathematical expressions with precision tracking |
| **Computational Biology** | `protocols` | id, name, organism, reagents, conditions, yield, source_file | Experimental procedures with reproducibility metadata |
| **Machine Learning** | `experiments` | id, model, dataset, hyperparams, metrics, baseline_delta, source_file | Training runs with performance deltas |
| **Drug Discovery** | `compounds` | id, smiles, target, binding_affinity, toxicity_flag, synthesis_route | Chemical entities with pharmacological properties |
| **Software Architecture** | `components` | id, name, api_surface, dependencies, performance_profile | System components with interface contracts |
| **Mathematics** | `conjectures` | id, statement, known_cases, approaches_tried, current_status | Open problems with approach history |
| **Climate Science** | `simulations` | id, model, resolution, forcing_scenario, key_outputs, validation_against | Model runs with validation metadata |

### 3c. Adding Custom Entity Types

To add a domain-specific entity type, append to the `entity_types` section of the schema:

```yaml
  # Domain-specific type
  experiments:
    universal: false
    description: "ML training runs with configuration and results"
    authority: calculator
    fields:
      id:
        type: string
        required: true
      model_architecture:
        type: string
        required: true
      dataset:
        type: string
        required: true
      hyperparameters:
        type: string
        required: true
        description: "Key hyperparameters as structured text"
      metrics:
        type: string
        required: true
        description: "Primary evaluation metrics and values"
      baseline_comparison:
        type: string
        required: false
      session:
        type: string
        required: true
      source_file:
        type: string
        required: true
```

### 3d. Constraint Category Configuration

Add domain-specific constraint category prefixes:

| Domain | Added Categories |
|:-------|:----------------|
| **Theoretical Physics** | `P` (Perturbative), `B` (BCS/Condensate), `T` (Topological), `N` (NCG Axiom) |
| **Machine Learning** | `A` (Architectural), `C` (Convergence), `G` (Generalization), `E` (Efficiency) |
| **Drug Discovery** | `B` (Binding), `T` (Toxicity), `PK` (Pharmacokinetic), `SY` (Synthesis) |
| **Software Architecture** | `P` (Performance), `SC` (Scalability), `SE` (Security), `C` (Compatibility) |
| **Climate Science** | `R` (Resolution), `P` (Parameterization), `V` (Validation), `SC` (Scenario) |

## 4. Knowledge Index Format

The knowledge index is the canonical machine-readable artifact. One JSON file. One source of truth.

### 4a. Index Structure

```json
{
  "metadata": {
    "schema_version": "1.0",
    "project": "{project-name}",
    "domain": "{domain}",
    "created": "{ISO-8601}",
    "last_updated": "{ISO-8601}",
    "entity_types": ["sessions", "constraints", "gates", "..."],
    "total_entities": 0,
    "sessions_indexed": 0
  },
  "sessions": [],
  "constraints": [],
  "gates": [],
  "proven_results": [],
  "closed_approaches": [],
  "active_channels": [],
  "confidence_trajectory": [],
  "data_provenance": [],
  "references": []
}
```

Domain-specific entity types appear as additional top-level arrays:

```json
{
  "metadata": { "entity_types": ["sessions", "...", "experiments"] },
  "sessions": [],
  "experiments": []
}
```

### 4b. Index Rules

1. **Single source of truth** — all queries resolve against this file
2. **Single writer** — only the indexer agent (or `/weave --update`) mutates this file
3. **Append-oriented** — entities are added or updated, never silently deleted
4. **Schema-validated** — every entry must have all `required: true` fields from the schema
5. **Source-traced** — every entry must have a `source_file` pointing to its origin

### 4c. Deduplication Rules

When the same entity appears in multiple source files:

1. **Within an entity type**: Same `id` in multiple entries → keep the one from the highest-authority source
2. **Cross-session updates**: Later session data supersedes earlier session data for the same entity
3. **Authority hierarchy**: When in doubt, follow the `source_authority` list in the schema
4. **Curated fields preserved**: If a previous entry has manually curated fields (names, notes, errata), preserve them during updates

## 5. Indexer Maintenance Protocol

This is the operational specification for how the indexer agent maintains the knowledge index without scripts. It replaces `extract_entities.py`.

### 5a. When to Update

| Trigger | What the Indexer Does |
|:--------|:------------------------|
| Session complete | Read synthesis file + gate verdicts → extract new entries for all applicable entity types |
| `/weave --update` invoked | Scan all session files for new or changed content → incremental index update |
| New researcher folder created | Add entry to `references` array |
| Constraint map updated | Extract new constraint entries into `constraints` array |
| Data computation completed | Add `data_provenance` entry linking script → outputs → gates |

### 5b. Extraction Protocol (Per Session)

When a session completes and the indexer updates the index:

1. **Read the schema** — load `tools/knowledge-schema.yaml` to know what entity types exist and what fields are required
2. **Read session artifacts** — synthesis file (highest priority), gate verdicts, session minutes
3. **For each entity type in the schema**:
   - Scan the session artifacts for content matching this entity type's `description`
   - For each entity found, populate all `required: true` fields
   - Set `source_file` to the artifact where the entity was found
   - If the entity already exists in the index (by `id`), update it (respecting authority hierarchy)
   - If the entity is new, append it to the appropriate array
4. **Update metadata** — increment `sessions_indexed`, update `last_updated`, recalculate `total_entities`
5. **Validate** — confirm all required fields are present, all enum values are valid, all source_files exist

### 5c. What the Indexer Reads (Priority Order)

The indexer reads session artifacts in this order. Higher-priority sources override lower when conflicts exist.

| Priority | Source Pattern | What It Contains |
|:---------|:-------------|:-----------------|
| 1 (highest) | `*sagan-verdict*` / `*skeptic-verdict*` | Confidence trajectory, gate verdicts |
| 2 | `*synthesis*` | Session outcomes, closed approaches, active channels |
| 3 | `*gate_verdicts*` | Gate pass/fail/closed classifications |
| 4 | `session*.md` | Session minutes, agent reports, decisions |
| 5 (lowest) | `data/output/*` | Raw computation outputs, data files |

### 5d. What the Indexer Does NOT Do

- **Does NOT interpret content** — indexes what files say, never assesses significance
- **Does NOT classify gate verdicts** — that's the skeptic's authority
- **Does NOT write constraint map entries** — that's the coordinator's authority
- **Does NOT run computation scripts** — only reads their outputs
- **Does NOT auto-fix inconsistencies** — reports them, lets the appropriate agent resolve
- **Does NOT curate names or add commentary** — preserves source language exactly

### 5e. Incremental vs Full Rebuild

| Mode | When | How |
|:-----|:-----|:----|
| **Incremental** | After each session | Read only new/changed files, update affected entries |
| **Full rebuild** | On `/weave --update --full` or when index is corrupted | Read ALL session artifacts, rebuild index from scratch, preserve curated fields |

For full rebuild, the indexer:
1. Backs up curated fields (anything manually added: names, errata, notes)
2. Reads every session directory in order
3. Rebuilds all entity arrays from source files
4. Restores curated fields by matching on entity `id`
5. Reports statistics: entities added, updated, removed

## 6. Query Interface — The `/weave` Skill

The `/weave` skill is the query interface to the knowledge index. It is schema-aware: it reads the schema to know what entity types and fields are available.

### 6a. Core Queries (No Python Required)

These queries work by reading `knowledge-index.json` directly:

| Command | What It Does |
|:--------|:-------------|
| `/weave --show {type}` | Render all entries of an entity type as a markdown table |
| `/weave --show all` | Summary statistics for all entity types |
| `/weave --search "term"` | Case-insensitive search across all entity fields |
| `/weave --trace {entity-id}` | Follow the full evidence chain for an entity (source files, related gates, provenance) |
| `/weave --validate` | Check index against schema (missing required fields, invalid enums, orphaned references) |
| `/weave --update` | Trigger indexer to scan for new session content and update index |
| `/weave --update --full` | Full index rebuild from all session artifacts |
| `/weave --schema` | Display the current knowledge schema (entity types, fields, categories) |
| `/weave --stats` | Index statistics (entity counts, sessions indexed, last update) |

### 6b. Provenance Queries

| Command | What It Does |
|:--------|:-------------|
| `/weave --provenance {filename}` | Show the script → input → output → gate chain for a data file |
| `/weave --gates` | Summary table of all gates with verdicts |
| `/weave --trajectory` | Confidence evolution timeline |
| `/weave --channels` | Active channels with survived constraints |

### 6c. Accelerated Queries (Python Required)

When the optional Python accelerator is installed:

| Command | What It Does | Requires |
|:--------|:-------------|:---------|
| `/weave --db-sync` | Rebuild SQLite FTS5 database from JSON index | `knowledge_db.py` |
| `/weave --db-search "term"` | BM25-ranked full-text search | `knowledge_db.py` |
| `/weave --graph` | Generate knowledge topology PNG | `visualize_knowledge.py` |
| `/weave --timeline` | Generate confidence timeline PNG | `visualize_knowledge.py` |
| `/weave --viz-all` | Generate all visualization artifacts | `visualize_knowledge.py` |

### 6d. Schema-Driven Query Resolution

When `/weave --show {type}` is invoked:

1. Read `tools/knowledge-schema.yaml`
2. Find the entity type definition
3. Read `tools/knowledge-index.json`
4. Extract the corresponding array
5. Render as a markdown table using the field definitions from the schema
6. Include column headers from field `description` values

This means **any entity type** added to the schema is automatically queryable — no skill code changes needed.

## 7. Authority Model Integration

The schema's `authority` field on each entity type connects to the agent archetype system. This prevents unauthorized writes.

### 7a. Authority Rules

| Entity Type | Authority | What This Means |
|:------------|:----------|:----------------|
| `sessions` | coordinator | Coordinator writes session metadata during synthesis |
| `constraints` | coordinator | Coordinator writes constraint entries (skeptic classifies, coordinator records) |
| `gates` | skeptic | Skeptic defines gate criteria and classifies verdicts |
| `proven_results` | workhorse | Workhorse verifies structural results (coordinator records after verification) |
| `closed_approaches` | coordinator | Coordinator records closures from gate verdicts |
| `active_channels` | coordinator | Coordinator tracks surviving approaches |
| `confidence_trajectory` | skeptic | Skeptic alone produces confidence estimates |
| `data_provenance` | indexer | Indexer tracks computation lineage |
| `references` | indexer | Indexer maintains reference corpus inventory |

### 7b. How Authority Works in Practice

Authority determines who **produces** the content, not who writes the JSON. The indexer is always the sole writer of `knowledge-index.json`. Authority controls whose session artifacts the indexer trusts for each entity type:

- For `gates` (authority: skeptic): the indexer extracts gate entries from skeptic verdict files, not from session minutes
- For `constraints` (authority: coordinator): the indexer extracts from the constraint map, which only the coordinator writes
- For `confidence_trajectory` (authority: skeptic): the indexer reads only skeptic-authored confidence estimates

### 7c. Override Protocol

If an entity needs updating by someone other than the authority:
1. The requesting agent flags the issue in a session
2. The authority agent reviews and approves the change
3. The coordinator records the change with provenance
4. The indexer updates the index entry

## 8. Relationship to Constraint Map

The knowledge index and the constraint map are complementary systems with distinct roles.

### 8a. How They Differ

| Aspect | Constraint Map | Knowledge Index |
|:-------|:---------------|:----------------|
| **Format** | Markdown (human-readable) | JSON (machine-queryable) |
| **Writer** | Coordinator | Indexer |
| **Purpose** | Logic backbone — defines solution space boundaries | Institutional memory — everything the project has established |
| **Scope** | Constraints, theorems, channels, gates, benchmarks | All entity types including sessions, provenance, references |
| **Query** | Cite by ID in prose | `/weave` skill queries |

### 8b. How They Connect

The constraint map **feeds into** the knowledge index. The indexer extracts structured entries from the constraint map into the JSON index:

| Constraint Map Section | → Knowledge Index Type |
|:-----------------------|:-----------------------|
| Constraint entries (all categories) | `constraints` + `closed_approaches` |
| Structural Theorems (ST-NN) | `proven_results` |
| Active Channels (AC-NN) | `active_channels` |
| Unvalidated Gates | `gates` (verdict: UNVALIDATED) |
| Gate verdicts (PASS/CLOSED) | `gates` (verdict updated) |
| Observational Benchmarks | Stored as `constraints` with category `O` |

### 8c. Constraint Map Spec

The full constraint map specification (entry schema, state machine, category taxonomy, authority model, framing rules, bootstrap template) lives at `agent-memory/constraint-map.md` (in the plugin root). That document is the companion to this one — it specifies the *logic* of constraints; this document specifies the *storage and query* infrastructure.

## 9. Bootstrap — Empty Index Generation

When a new project is created, the indexer generates an empty index from the schema.

### 9a. Generation Algorithm

```
1. Read tools/knowledge-schema.yaml
2. Create JSON object with:
   a. metadata block (project name, domain, date, entity_types list)
   b. One empty array for each entity type in the schema
3. Write to tools/knowledge-index.json
4. Verify: every entity type in schema has a corresponding array in JSON
```

### 9b. Empty Index Template

```json
{
  "metadata": {
    "schema_version": "1.0",
    "project": "{project-name}",
    "domain": "{domain}",
    "created": "{today}",
    "last_updated": "{today}",
    "entity_types": [
      "sessions",
      "constraints",
```

## 10. Optional Python Accelerator

For projects that outgrow pure JSON queries, the Python accelerator provides SQLite full-text search and automated visualization.

### 10a. When to Install

| Project Size | Recommendation |
|:-------------|:---------------|
| < 10 sessions, < 500 entities | JSON-only (no Python needed) |
| 10-50 sessions, 500-5,000 entities | Install `knowledge_db.py` for ranked search |
| 50+ sessions, 5,000+ entities | Full accelerator (database + visualization) |

### 10b. What Gets Installed

| File | Lines | Purpose | Dependencies |
|:-----|:------|:--------|:-------------|
| `knowledge_db.py` | ~520 | SQLite + FTS5 sync and query | Python 3.8+, sqlite3 (stdlib) |
| `visualize_knowledge.py` | ~1,000 | PNG graphs + Mermaid diagrams | matplotlib, networkx |

### 10c. Schema-Aware Accelerator

The accelerator tools read the schema to know what tables to create and what fields to index:

```
knowledge_db.py:
  1. Read tools/knowledge-schema.yaml
  2. For each entity type → CREATE TABLE with schema-defined fields
  3. CREATE VIRTUAL TABLE fts USING fts5() across all text fields
  4. INSERT all entries from knowledge-index.json
  5. Provide BM25-ranked search via --search
```

This means the accelerator is **automatically domain-adapted** — it creates tables for whatever entity types the schema defines, including domain-specific ones.

### 10d. Generalized Visualization

The visualization tool reads the schema to determine:
- What entity types to include in the knowledge graph
- What relationships exist between entities (gates → constraints, provenance chains)
- What timeline data to plot (confidence_trajectory)
- What summary tables to generate (gates by verdict, constraints by category)

Domain-specific milestones and annotations are read from a `viz_config` section in the schema (optional):

```yaml
viz_config:
  milestones:
    - session: "3"
      label: "First gate passed"
      color: green
    - session: "8"
      label: "Major constraint discovered"
      color: red
  entity_colors:
    proven_results: green
    closed_approaches: red
    gates: orange
    active_channels: yellow
    sessions: blue
```

## 11. Migration Path (Origin Project)

For the origin project (28-session phonon-exflation), the migration from script-driven to schema-driven is:

1. **Generate schema** from existing `extract_entities.py` patterns → `knowledge-schema.yaml`
2. **Preserve existing index** — the current `knowledge-index.json` is already valid; just add metadata
3. **Make accelerator schema-aware** — update `knowledge_db.py` to read entity types from schema instead of hardcoded tables
4. **Keep `extract_entities.py` as a legacy tool** — it works; no need to delete it. New projects don't ship it.

The schema captures what the regex patterns *mean*, not how they *match*. A indexer agent reading a synthesis file and following the schema extracts the same information the regex would — but for any domain, not just physics.
